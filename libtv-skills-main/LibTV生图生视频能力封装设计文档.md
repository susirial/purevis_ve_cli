# LibTV 生图/生视频能力封装设计文档

## 1. 背景

当前 `libtv-skill` 的本质是一个面向 Agent 的会话型适配层。它已经具备这些底层能力：

- 创建会话
- 发送自然语言创作请求
- 上传图片或视频素材
- 查询会话进展
- 下载最终结果

但从产品化和平台化视角看，这一层仍然偏“协议驱动”，而不是“能力驱动”。上层调用方如果直接使用它，会面临几个问题：

- 对外暴露的是会话概念，而不是生图、生视频两个明确能力
- 输入是自由文本，会导致模型、时长、画幅、输出约束不稳定
- 轮询、结果判断、失败重试要由每个调用方重复实现
- 无法形成统一的计费、监控、幂等和 SLA 管理

因此，需要在现有技能之上增加一层 **能力封装层**，将底层会话能力收敛为两个清晰、可治理、可演进的产品能力：

1. 生图能力
2. 生视频能力

---

## 2. 设计目标

### 2.1 目标

本次封装目标如下：

- 对外提供稳定的“能力接口”，而不是暴露底层会话脚本
- 强约束模型选择，确保只走指定模型集合
- 将结构化参数转换为稳定的底层调用协议
- 统一任务状态、结果结构、错误码和重试策略
- 满足业界最佳实践：异步任务化、幂等、可观测、可扩展

### 2.2 非目标

本次不做以下内容：

- 不重新实现底层生成引擎
- 不替代 LibTV 后端的创作编排 Agent
- 不在用户侧做复杂 prompt 工程
- 不把所有图片编辑、视频编辑、视频续写能力一次性都产品化

本阶段只聚焦两个一等能力：`generate_image` 和 `generate_video`。

---

## 3. 模型范围与能力定位

### 3.1 生图模型

生图能力仅允许使用以下模型：

- `lib_nano_2`
- `lib_nano_pro`

模型特性建议定义为：

| 模型 | 预估耗时 | 成本定位 | 能力定位 | 建议用途 |
|------|----------|----------|----------|----------|
| `lib_nano_2` | 约 25 秒 | 较低 | 标准质量、性价比高 | 大盘默认、探索式生成、批量试稿 |
| `lib_nano_pro` | 约 50 秒 | 较高 | 最强质量 | 定稿、品牌视觉、对细节要求高的任务 |

### 3.2 生视频模型

生视频能力仅允许使用以下模型：

- `seedance_2_0`
- `seedance_2_0_fast`
- `kling_o3`

模型特性建议定义为：

| 模型 | 速度特征 | 时长范围 | 能力定位 | 建议用途 |
|------|----------|----------|----------|----------|
| `seedance_2_0` | 较慢 | 4~15 秒 | 高质量主力模型 | 正式产出、质量优先 |
| `seedance_2_0_fast` | 较快 | 4~15 秒 | 快速预览模型 | 打样、快速试错、低等待场景 |
| `kling_o3` | 中等 | 4~15 秒 | 备选主力模型 | AB 对比、特殊风格尝试、模型兜底 |

说明：

- `seedance_2_0` 适合“质量优先”
- `seedance_2_0_fast` 适合“时延优先”
- `kling_o3` 不建议默认全量切入，建议作为补充路由和回退通道

## 4. 总体设计思路

### 4.1 封装原则

建议采用 **能力层 + 编排层 + 适配层** 三层结构：

1. **能力层**
   对外暴露生图、生视频两个清晰能力接口
2. **编排层**
   负责参数校验、模型策略、素材上传、状态管理、轮询、结果归一化
3. **适配层**
   复用现有 LibTV 会话技能，负责真正的会话创建、消息发送、查询和下载

### 4.2 为什么不能直接暴露底层会话

如果直接暴露底层会话，会出现以下问题：

- 业务方必须理解 `sessionId`、`projectUuid`、轮询细节
- 业务方会直接拼自然语言消息，导致调用风格不一致
- 不同业务方会自行设计状态机，最终难以治理
- 很难做统一的模型白名单和能力审计

因此，对外接口必须是能力语义，而内部再映射到会话语义。

---

## 5. 对外能力定义

建议把对外能力收敛为两个异步任务接口。

### 5.1 生图能力

能力名建议：

- `generate_image`

语义：

- 接收一段用户创作需求，可带参考图，异步返回图片结果

### 5.2 生视频能力

能力名建议：

- `generate_video`

语义：

- 接收一段用户创作需求，可带参考图或参考视频，异步返回视频结果

### 5.3 统一任务制

无论生图还是生视频，都应采用统一任务模型：

- 创建任务
- 返回任务 ID
- 后台异步执行
- 调用方轮询任务状态
- 成功后获取标准化结果

这样做的好处：

- 便于横向扩展到编辑、续写、转绘等能力
- 便于做队列治理和 SLA 管理
- 便于接入回调、事件总线、审计日志


### 6.2 创建生视频任务

## 7. 请求字段设计原则

### 7.1 `prompt` 必须保留用户原意

底层技能非常强调“用户侧只传话，不替用户创作”。因此封装层应遵守：

- 保留用户原始 `prompt`
- 不做文学化改写
- 不做风格脑补
- 不自行扩写成冗长 prompt

封装层允许做的只有两类增强：

1. 结构化参数拼装
2. 安全、合法性和边界约束补充

例如可以把 `aspect_ratio`、`duration_sec`、模型名等信息拼成稳定的附加说明，但不应该替用户发明内容。

### 7.2 `model` 应显式可控

由于各模型在成本、时延、质量上的差异明显，`model` 不应完全隐藏。最佳实践是：

- 对外允许显式指定模型
- 对外同时提供默认策略
- 默认策略可按“快 / 稳 / 质量优先”自动映射

例如：

- 生图默认 `lib_nano_2`
- 高质量生图默认 `lib_nano_pro`
- 生视频默认 `seedance_2_0_fast`
- 高质量生视频默认 `seedance_2_0`

### 7.3 `references` 统一抽象

无论生图还是生视频，都建议统一使用 `references` 数组，而不是零散参数。这样做有几个好处：

- 同时兼容本地路径和远程 URL
- 同时兼容图片和视频
- 为未来扩展多素材、多角色输入留空间

### 7.4 `project_policy` 必须显式化

建议提供两种策略：

- `reuse`
- `new`

语义如下：

- `reuse`：尽量复用当前项目上下文
- `new`：先切换到新项目，再发起任务

这比把是否切项目埋在内部规则里更容易被业务理解和控制。

## 8. 模型选择策略

### 8.1 生图策略

建议的生图默认策略如下：

| 场景 | 默认模型 | 原因 |
|------|----------|------|
| 默认单次生成 | `lib_nano_2` | 时延和成本更友好，适合作为大盘入口 |
| 多轮探索、批量试稿 | `lib_nano_2` | 便于快速收敛方向 |
| 需要高质量定稿 | `lib_nano_pro` | 质量上限更高 |
| 品牌物料、营销主视觉 | `lib_nano_pro` | 细节稳定性更重要 |

建议不要做的事：

- 不要“总是自动升到最强模型”
- 不要把低成本模型和高质量模型混成一个无差别能力

最佳实践是：**默认用性价比模型，允许显式升配。**

### 8.2 生视频策略

建议的生视频默认策略如下：

| 场景 | 默认模型 | 原因 |
|------|----------|------|
| 快速预览 | `seedance_2_0_fast` | 等待时间更短 |
| 正式成片 | `seedance_2_0` | 质量优先 |
| 模型对比或补充尝试 | `kling_o3` | 作为差异化通道 |
| 时延敏感交互场景 | `seedance_2_0_fast` | 响应更友好 |

### 8.3 是否自动回退

建议支持“有限回退”，但不要做无边界自动漂移。

推荐策略：

- 如果用户显式指定了模型，则默认 **不自动改模型**
- 如果用户使用的是默认策略，则可以在任务失败时按白名单回退一次

例如：

- 默认视频策略失败：`seedance_2_0_fast -> kling_o3`
- 默认高质量视频策略失败：`seedance_2_0 -> kling_o3`

不建议：

- 失败后悄悄切到任意模型
- 在用户无感知情况下改变成本档位

## 9. 状态机设计

能力封装层应定义统一状态，而不是把底层消息状态直接暴露给业务。

建议状态集：

- `accepted`
- `queued`
- `running`
- `succeeded`
- `failed`
- `timeout`
- `canceled`

状态语义建议：

| 状态 | 含义 |
|------|------|
| `accepted` | 请求已接收并完成参数校验 |
| `queued` | 已提交到后台等待执行 |
| `running` | 已创建会话并开始生成 |
| `succeeded` | 已拿到可交付结果 |
| `failed` | 明确失败，且不可自动恢复 |
| `timeout` | 超过预设等待时间仍未完成 |
| `canceled` | 被业务主动取消 |

说明：

- 底层会话接口没有天然任务状态机
- 上层必须根据“提交成功、轮询消息、是否提取到结果 URL、是否超时”等事件组合出统一状态

## 10. 内部编排流程

### 10.1 生图任务执行流程

1. 接收结构化请求
2. 校验 `prompt`、模型、画幅、素材类型
3. 如 `project_policy=new`，先切换项目
4. 如存在本地参考图，先上传并获得可访问地址
5. 将用户原始 `prompt` 与结构化约束拼接为标准请求消息
6. 创建会话并发送消息
7. 保存 `job_id`、`session_id`、`project_id`
8. 后台轮询会话消息
9. 识别图片结果
10. 下载并归档
11. 返回标准化结果

### 10.2 生视频任务执行流程

1. 接收结构化请求
2. 校验 `prompt`、模型、时长、画幅、素材类型
3. 校验 `duration_sec` 是否处于 4~15 秒之间
4. 如 `project_policy=new`，先切换项目
5. 如存在本地参考图或视频，先上传并获得可访问地址
6. 将用户原始 `prompt` 与结构化约束拼接为标准请求消息
7. 创建会话并发送消息
8. 保存 `job_id`、`session_id`、`project_id`
9. 后台轮询会话消息
10. 识别视频结果
11. 下载并归档
12. 返回标准化结果

## 11. Prompt 拼装策略

这是本设计里最容易走偏的部分。

### 11.1 原则

拼装目标不是“帮用户写 prompt”，而是“把结构化参数稳定地传给后端”。

因此建议消息模板采用以下风格：

```text
用户需求：
{prompt}

结构化要求：
- 能力：生图 / 生视频
- 指定模型：{model}
- 画幅比例：{aspect_ratio}
- 视频时长：{duration_sec} 秒
- 参考素材：{uploaded_reference_urls}
```

说明：

- `prompt` 原文保留
- 结构化约束单独列出
- 不做语言润色
- 不增加审美化描述

### 11.2 为什么这样是最佳实践

因为底层 LibTV 能力本质上还是自然语言驱动。完全不拼结构化信息会导致：

- 模型不可控
- 时长不可控
- 参考素材使用不稳定

但如果拼得过度，又会违反“用户侧不代替创作”的原则。

因此，**最优解是做参数注入，不做内容创作。**

## 12. 结果归一化设计

对外不应直接返回底层消息数组，而应返回稳定结果对象。

建议统一输出结构：

```json
{
  "job_id": "job_xxx",
  "status": "succeeded",
  "capability": "generate_image",
  "model": "lib_nano_pro",
  "session_id": "xxx",
  "project_id": "xxx",
  "project_url": "https://...",
  "outputs": [
    {
      "media_type": "image",
      "url": "https://...",
      "local_path": "/tmp/xxx.png",
      "width": 1024,
      "height": 1024
    }
  ],
  "metrics": {
    "queue_ms": 0,
    "generate_ms": 42000,
    "download_ms": 3200,
    "total_ms": 45200
  }
}
```

### 12.1 统一返回的意义

这样设计后：

- 业务不需要理解底层消息内容
- 后续可平滑接入更多模型
- 监控、计费、审计都可以基于统一字段建设

## 13. 幂等与重试

这部分是生产可用性的关键。

### 13.1 幂等键

建议创建任务时支持 `request_id`，作为幂等键使用。

规则建议：

- 相同 `request_id` + 相同能力类型 + 相同核心参数
- 在幂等窗口内只创建一次任务

这样可以避免：

- 前端超时重试导致重复生成
- 用户多次点击提交导致重复扣费

### 13.2 重试策略

建议区分三类重试：

1. **上传重试**
   网络失败时可重试 1~2 次
2. **轮询重试**
   查询失败可做指数退避
3. **模型回退重试**
   仅对默认策略任务执行一次白名单内回退

不建议：

- 对所有失败无限重试
- 对用户显式指定模型擅自回退

---

## 14. 超时与 SLA 设计

由于模型耗时差异大，建议分能力定义超时策略。

### 14.1 建议超时

| 能力 | 默认等待阈值 | 建议说明 |
|------|--------------|----------|
| 生图 `lib_nano_2` | 90 秒 | 正常应在较短时间返回 |
| 生图 `lib_nano_pro` | 150 秒 | 允许更长计算时间 |
| 生视频 `seedance_2_0_fast` | 300 秒 | 视频生成天然更慢 |
| 生视频 `seedance_2_0` | 420 秒 | 质量模型需要更长窗口 |
| 生视频 `kling_o3` | 360 秒 | 作为中间档默认值 |

### 14.2 超时策略

超时后建议：

- 任务状态置为 `timeout`
- 返回当前 `project_url`
- 允许调用方后续继续手动查询

这样既不会无限阻塞，也不会让任务彻底失联。

## 15. 错误码设计

建议统一错误码，而不是直接透传底层异常文本。

建议错误码分层：

- `INVALID_ARGUMENT`
- `UNSUPPORTED_MODEL`
- `INVALID_DURATION`
- `INVALID_REFERENCE_TYPE`
- `UPLOAD_FAILED`
- `CREATE_SESSION_FAILED`
- `POLLING_FAILED`
- `RESULT_NOT_FOUND`
- `TIMEOUT`
- `INTERNAL_ERROR`

最佳实践：

- 对外返回稳定错误码
- 对内保留底层错误原文，便于排障

## 16. 可观测性设计

封装层上线后，必须具备以下观测指标：

### 16.1 指标

- 请求量
- 成功率
- 平均生成耗时
- 各模型成功率
- 各模型 P95 耗时
- 上传失败率
- 超时率
- 自动回退触发率

### 16.2 日志

建议记录：

- `job_id`
- `request_id`
- `capability`
- `model`
- `session_id`
- `project_id`
- 状态流转
- 上传素材数量
- 轮询次数
- 最终结果数量

### 16.3 链路追踪

如果能力层以服务方式提供，建议引入 trace_id，贯穿：

- 接口请求
- 素材上传
- 会话创建
- 轮询查询
- 下载归档

## 17. 安全与治理

### 17.1 模型白名单

必须在封装层强制白名单，只允许以下模型：

- 生图：`lib_nano_2`、`lib_nano_pro`
- 生视频：`seedance_2_0`、`seedance_2_0_fast`、`kling_o3`

### 17.2 参数治理

建议约束：

- 视频时长必须在 `4~15` 秒
- 图片能力拒绝视频素材
- 视频能力允许图片或视频素材
- 限制单次素材数量
- 限制单次输出数量

### 17.3 成本治理

建议加入以下控制：

- 默认模型策略
- 租户级模型权限
- 高成本模型审批或配额
- 单日调用额度限制

## 18. 推荐产品策略

如果从产品落地角度设计，建议不是只暴露“模型名”，而是暴露“能力档位 + 模型覆盖”的双层方式。

### 18.1 生图档位

- `draft` -> 默认映射 `lib_nano_2`
- `quality` -> 默认映射 `lib_nano_pro`

### 18.2 生视频档位

- `fast_preview` -> 默认映射 `seedance_2_0_fast`
- `quality` -> 默认映射 `seedance_2_0`
- `alternative` -> 默认映射 `kling_o3`

### 18.3 为什么要做双层

这样设计的好处：

- 产品层对用户讲“快 / 质量 / 备选”
- 技术层仍然保留明确模型控制
- 未来更换底层模型时，对外接口更稳定

## 19. 最小可行实现建议

建议按以下顺序落地：

### Phase 1：能力封装 MVP

- 封装 `generate_image`
- 封装 `generate_video`
- 接入统一任务状态机
- 支持本地素材上传
- 支持结果下载
- 支持显式模型指定

### Phase 2：生产增强

- 幂等键
- 自动回退
- 超时治理
- 监控与日志
- 配额与权限控制

### Phase 3：高级能力

- Webhook / 回调
- 批量任务
- 编辑类能力扩展
- 多模型 AB 对比模式

## 20. 推荐方案总结

本次封装建议采用以下核心决策：

1. **对外只暴露两个能力**
   `generate_image` 和 `generate_video`
2. **对外采用异步任务模型**
   创建任务、查询状态、获取结果
3. **对内复用现有会话技能**
   不推翻当前实现，只在上层做能力编排
4. **显式模型白名单**
   生图仅 `lib_nano_2` / `lib_nano_pro`，生视频仅 `seedance_2_0` / `seedance_2_0_fast` / `kling_o3`
5. **保留用户原始 prompt**
   只拼参数，不代替创作
6. **统一结果对象和状态机**
   便于平台治理和后续扩展
7. **默认策略优先，显式覆盖保留**
   满足成本、时延和质量平衡

---

## 21. 一句话结论

最佳实践不是把 `libtv-skill` 原样暴露给业务，而是把它升级为一个 **能力化、任务化、可治理的媒体生成封装层**：对外只有“生图”和“生视频”，对内再利用现有会话机制完成上传、生成、轮询和结果归档。
