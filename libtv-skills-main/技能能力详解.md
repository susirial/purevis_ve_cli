# LibTV Skills 能力详解

## 1. 文档目的

这份文档从“能力”而不是“源码位置”的角度，系统说明当前仓库提供了什么、适合解决什么问题、如何组合调用、有哪些输入输出约定，以及代码真实行为和说明文档之间存在哪些需要注意的差异。

结论先行：当前仓库只提供 **1 个核心技能入口**，名称为 `libtv-skill`。它本质上不是一个“直接生成图片/视频的模型”，而是一个 **面向 Agent 的会话编排适配层**。它负责把用户的自然语言创作需求、参考图片/视频、会话上下文和结果下载流程串起来，并通过 LibTV 的 OpenAPI 调用后端创作能力。

---

## 2. 仓库整体能力概览

### 2.1 核心定位

这个技能包面向支持 OpenClaw 规范的 AI Agent 平台，作用是让 Agent 具备以下能力：

1. 以自然语言形式提交创作任务
2. 创建或复用创作会话
3. 查询任务进展和返回消息
4. 上传图片或视频作为参考素材
5. 切换创作项目上下文
6. 提取并下载最终生成结果

### 2.2 能解决的问题

从技能描述和实现来看，它覆盖的是一类很宽的“多模态创作请求”，包括但不限于：

- 文生图
- 文生视频
- 图生视频
- 视频续写
- 图片或视频编辑
- 元素替换
- 局部修改
- 风格迁移
- 分镜图生成
- 短剧、短漫、MV、TVC、宣传片等复杂内容生成

也就是说，这个技能并不要求调用方先把任务拆得很细。调用方通常只需要把用户原始意图发送出去，后端系统再决定模型、工作流和执行细节。

---

## 3. 最重要的设计原则

### 3.1 用户侧 Agent 的职责非常克制

这个仓库反复强调一个核心原则：

> 用户侧 Agent 只负责“传话、上传、取回结果”，不负责替用户做创作。

具体来说：

- 用户给了文本需求，就原样发送
- 用户给了本地参考素材，就先上传，再把素材地址拼进请求
- 任务完成后，再把结果取回并展示

### 3.2 不建议调用方做的事

按技能说明，调用方不应该：

- 替用户润色 prompt
- 擅自翻译或扩写用户意图
- 把一个复杂请求拆成多次独立创作
- 自己推演剧情、镜头、风格细节
- 用主观猜测补充“电影级光影”“8K”“超写实”等描述

这个原则非常关键，因为仓库作者的假设是：**后端 Agent 比前端调用方更懂如何选模型和编排创作链路**。因此，前端越少“自作主张”，整体生成质量越稳定。

---

## 4. 技能入口的触发语义

### 4.1 触发范围非常宽

从技能元数据来看，只要用户表达的是“图片或视频创作/编辑相关需求”，就应该触发这个技能。典型触发意图包括：

- “画一只猫”
- “做一个海报”
- “生成一段动漫视频”
- “把这个视频里的纸船换成爱心”
- “参考这张图做一个产品广告片”
- “帮我复刻这个视频的风格”
- “用这首歌生成 MV”
- “一句话生成短剧”
- “帮我看看现在生成到哪一步了”

### 4.2 触发信号不仅来自创作指令

除了直接创作类描述，以下情况也应视为触发：

- 用户提到 `liblib` 或 `libtv`
- 用户要上传参考图或参考视频
- 用户要求查看生成进度
- 用户希望下载生成结果

换句话说，这个技能不是只负责“开始生成”，而是覆盖 **创建任务到取回结果的完整会话生命周期**。

---

## 5. 运行依赖与环境约束

### 5.1 必要依赖

运行前至少需要满足两个条件：

- 系统可用 `python3`
- 环境变量中配置了 `LIBTV_ACCESS_KEY`

### 5.2 可选环境变量

还支持两个可选环境变量，用于覆盖服务地址：

- `OPENAPI_IM_BASE`
- `IM_BASE_URL`

如果都不提供，默认会连接到官方 IM 服务地址。

### 5.3 依赖特征

实现只依赖 Python 标准库，没有第三方包依赖。这意味着：

- 安装门槛低
- 便于在 Agent 环境直接执行
- 网络、超时、重试、上传等逻辑均为手写

---

## 6. 能力拆解：五个操作单元

虽然仓库只暴露一个技能入口，但它内部实际上由五个可组合的操作单元组成。对调用方来说，理解这五个单元比理解目录结构更重要。

### 6.1 `create_session`

#### 能力定位

用于：

- 创建一个新的创作会话
- 向已有会话追加一条自然语言消息
- 仅建立或绑定会话，不立即发送内容

#### 输入

- 可选消息文本
- 可选会话 ID

#### 输出

成功时会输出：

- `projectUuid`
- `sessionId`
- `projectUrl`

其中：

- `sessionId` 是后续轮询进度的关键标识
- `projectUuid` 是项目上下文标识
- `projectUrl` 是基于 `projectUuid` 拼接出来的画布地址，不是服务端直接返回的独立字段

#### 适用场景

- 首次提交“生图/生视频”需求
- 在同一任务里继续补充要求
- 建立一个后续可复用的任务上下文

#### 注意点

- 如果服务端没有返回 `sessionId`，脚本会直接报错退出
- 不传消息时，也可以拿到会话上下文

---

### 6.2 `query_session`

#### 能力定位

用于查询某个会话中的消息列表，通常被上层轮询逻辑拿来判断创作是否完成。

#### 输入

- 必填会话 ID
- 可选 `after-seq`，用于增量拉取
- 可选 `project-id`，用于在输出中附加 `projectUrl`

#### 输出

核心输出是：

- `messages`

当调用方额外传入 `project-id` 时，还会多返回：

- `projectUrl`

#### 适用场景

- 轮询任务进展
- 拉取新消息而不是全量消息
- 在任务完成时一并给出画布地址

#### 注意点

- `projectUrl` 不是查询接口天然返回的，而是脚本根据传入的 `project-id` 本地拼出来的
- 这个脚本本身只“查一次”，并不内置轮询循环
- 真正的轮询节奏需要调用方自己控制

---

### 6.3 `change_project`

#### 能力定位

用于把当前密钥所绑定的项目切换到新的项目上下文。

#### 输入

- 无业务参数

#### 输出

- `projectUuid`
- `projectUrl`

#### 适用场景

- 希望把新任务和旧任务隔离
- 明确要求“新建一个任务”
- 不希望后续会话继续落到旧项目中

#### 注意点

- 这是“切项目”，不是“切会话”
- 影响的是后续创作默认使用的项目上下文

---

### 6.4 `upload_file`

#### 能力定位

用于把本地图片或视频上传到 OSS，拿到一个后续可在会话消息中引用的可访问地址。

#### 输入

- 一个本地文件

#### 输出

- `url`

#### 适用场景

- 用户给了参考图，要求“按这个生成”
- 用户给了现有视频，要求“帮我修改”
- 用户给了素材，希望与文本要求组合使用

#### 注意点

- 本地会先检查是否是图片或视频类型
- 类型判断依赖 MIME 推断，不是内容级别校验
- 文档中声明文件大小需在 200MB 以内，但代码里没有看到本地大小检查，实际限制更可能由服务端兜底

---

### 6.5 `download_results`

#### 能力定位

用于从会话消息里提取图片/视频结果链接，并将结果并行下载到本地。

#### 输入

支持两种模式：

1. 传入会话 ID，让脚本自动查询并提取 URL
2. 直接传入一组 URL，跳过会话查询

还支持可选参数：

- 输出目录
- 文件名前缀
- 并发下载线程数

#### 输出

成功时主要返回：

- `output_dir`
- `downloaded`
- `total`

若有部分下载失败，还会附加：

- `errors`

#### 适用场景

- 任务完成后自动取回图片/视频
- 将结果批量保存到本地
- 直接处理已经拿到的一组资源地址

#### 注意点

- 它会同时从两类消息中提取结果：
- 一类是 `tool` 角色消息中的结构化 `task_result`
- 一类是 `assistant` 文本中的资源链接

另外还有几个实现层面的真实行为值得注意：

- 只匹配特定资源域名下的图片和视频后缀
- 会去重，保持首次出现顺序
- 默认并发线程数为 5
- 即使部分文件下载失败，也会输出成功下载结果和错误明细

## 7. 典型工作流

### 7.1 纯文本生成

这是最标准的使用方式：

1. 用 `create_session` 发送用户原始创作意图
2. 记录 `sessionId` 和 `projectUuid`
3. 用 `query_session` 轮询新消息
4. 发现结果链接后，用 `download_results` 取回结果
5. 最终向用户展示结果和项目画布入口

### 7.2 基于参考素材的编辑

当用户提供本地图片或视频并要求修改时：

1. 用 `upload_file` 上传素材
2. 拿到素材 URL
3. 把“用户原始指令 + 素材 URL”一起交给 `create_session`
4. 后续流程与纯文本生成相同

### 7.3 基于参考素材的再创作

当用户说“按这张图做一个新内容”时：

1. 先上传参考素材
2. 把参考地址拼入用户原始描述
3. 创建会话并发起任务
4. 轮询并下载结果

### 7.4 在已有任务中追加要求

如果用户是接着前面的任务继续修改：

1. 复用已有 `sessionId`
2. 用 `create_session` 追加新消息
3. 继续轮询
4. 下载新的结果

### 7.5 创建隔离任务

如果用户明确希望“新起一个项目”：

1. 先用 `change_project`
2. 再调用 `create_session`
3. 让新会话落在新的项目上下文中

## 8. 调用方需要自己承担的编排逻辑

### 8.1 轮询不是脚本自动完成的

仓库文档给出了推荐轮询策略，但这些策略并没有封装成单个自动执行的脚本。也就是说，真正负责“每隔多久查一次、什么时候结束、失败后怎么重试”的，是上层 Agent。

### 8.2 推荐策略来自技能说明，而不是强制代码

技能说明建议：

- 每 8 秒查询一次
- 首次增量查询用 `after-seq=0`
- 后续使用上次收到的最大 `seq`
- 超过约 3 分钟仍未完成时停止主动轮询
- 单次查询失败可重试 1 次
- 连续多次失败后向用户报错

但请注意，这些是 **编排建议**，不是脚本内部硬编码的强制规则。

### 8.3 完成判定也依赖上层理解消息

代码本身没有统一的“任务完成状态字段”。当前设计更偏向：

- 调用方读取 `messages`
- 在消息中寻找资源链接或结构化结果
- 以此判断任务是否已产出可交付内容

因此，如果要做成稳定的 Agent 能力，调用方必须具备一定的结果判定逻辑。

---

## 9. 输出契约与关键字段

### 9.1 会话级标识

最关键的两个标识是：

- `sessionId`：会话主键，用于继续对话和查询进度
- `projectUuid`：项目上下文标识，用于构造画布入口

### 9.2 展示级字段

- `projectUrl`：项目画布入口
- `url`：上传后返回的素材地址
- `downloaded`：已下载结果列表

### 9.3 一个容易误解的点

`projectUrl` 并不是所有接口都直接返回的真实服务端字段。至少从当前实现看，它在多个地方是由本地逻辑通过 `projectUuid` 拼接得到的。因此：

- 它是稳定可推导字段
- 但不应误以为这是服务端所有接口统一返回的原生字段

---

## 10. 实现层面的真实能力边界

### 10.1 这个仓库做的是“协议适配”，不是“创作决策”

从代码看，这个仓库没有做下面这些事：

- 模型选择
- 参数调优
- prompt 工程
- 剧情拆解
- 分镜编排
- 风格分析

它只是把请求可靠地送到后端，并把返回可靠地拿回来。

### 10.2 下载能力比表面描述更强

`download_results` 不只是“下载一组地址”，还承担了结果发现职责：

- 能解析 `tool` 消息中的结构化结果
- 能从 `assistant` 文本中抽取直链
- 能并行下载
- 能做基础命名和错误收集

这意味着它实际上是一个“小型结果归档器”。

### 10.3 上传能力比文档描述略弱

上传脚本做了基本 MIME 判断，但没有：

- 本地文件大小限制校验
- 内容签名级别校验
- 更细粒度的媒体合法性分析

所以它更像一个“轻量上传适配器”，不是严格的上传网关。

## 11. 文档与代码之间的差异

这一部分很重要，因为它直接影响集成正确性。

### 11.1 上传接口描述与代码实现不一致

说明文档中写的是上传接口为：

- `/openapi/file/upload`

但代码真实调用的是：

- `/openapi/upload`

如果以当前可执行行为为准，应优先相信代码实现。文档后续最好统一，否则会给二次集成者造成混淆。

### 11.2 200MB 限制只存在于说明层

技能说明提到上传文件大小需在 200MB 以内，但代码里没有看到本地大小校验。当前更准确的表述应该是：

- 文档声明有该限制
- 客户端未主动校验
- 真实限制可能由服务端执行

### 11.3 轮询规则主要是操作建议

技能说明写得像“标准流程”，但从实现上看：

- 没有自动轮询器
- 没有内置超时控制器
- 没有统一结果完成状态机

这些能力需要调用方自己补齐。

### 11.4 项目地址的来源容易被误读

说明中多次展示 `projectUrl`，容易让人以为它是接口原生字段。实际上当前实现是：

- 只要知道 `projectUuid`
- 就能在本地拼出 `projectUrl`

这是一个“派生字段”，不是必须依赖服务端返回。

---

## 12. 错误处理方式

### 12.1 总体风格

所有脚本的错误处理都比较直接：

- 打印错误信息到标准错误
- 退出进程

这适合 Agent 以子进程方式调用并捕获失败状态。

### 12.2 网络与接口错误

公共请求层统一处理两类问题：

- HTTP 错误
- 网络错误

处理方式是直接输出错误内容并退出，不做复杂恢复。

### 12.3 局部失败容忍

只有下载逻辑表现出更温和的失败模型：

- 某些文件下载失败不会阻断全部结果
- 会在最终输出里保留成功列表和错误列表

这对批量取回结果很实用。

## 13. 对接这个技能时的推荐理解方式

如果站在“平台集成者”视角，最好的理解方式不是把它看成五个孤立脚本，而是看成一个统一的能力闭环：

1. 接收用户意图
2. 如有素材，先上传
3. 创建或复用会话
4. 查询进度
5. 提取结果
6. 下载归档
7. 向用户交付结果与项目入口

换句话说，这个仓库真正交付的是：

> 一个适合 Agent 自动化调用的 LibTV 创作会话协议层。

## 14. 最终评估

### 14.1 优点

- 目标清晰，职责边界明确
- 以会话为中心，适合 Agent 编排
- 只依赖标准库，部署简单
- 支持上传、轮询、项目切换、下载等完整闭环
- 技能触发语义覆盖面广，适合作为统一入口

### 14.2 当前不足

- 上传接口说明与实现不一致
- 文件大小限制未在客户端落实
- 轮询策略只在文档层存在，没有封装成自动执行单元
- 完成态判断依赖上层调用方自行理解消息

### 14.3 最适合的使用方式

最适合把它接入一个具备任务编排能力的 Agent 中，由 Agent 来负责：

- 判断何时上传素材
- 维护 `sessionId`
- 进行增量轮询
- 判断何时完成
- 在结束时统一下载和展示结果

而不是把它当成一个“用户手动逐条执行命令的工具箱”。

---

## 15. 一句话总结

当前仓库的核心能力不是“生成图片或视频本身”，而是 **把用户的多模态创作需求可靠地转换为 LibTV 会话调用，并把结果完整取回**。它最像一个面向 AI Agent 的创作工作流桥接层。
