# 智能体模型与初始化说明文档

## 1. 文档目的

本文档用于说明系统中总控与各专业智能体的模型使用方式、初始化参数、装配结构、配置加载路径以及依赖注入机制，帮助维护者快速理解：

- 每个智能体在运行时依赖什么模型能力
- 哪些模型是在 Agent 层生效，哪些是在工具层生效
- 模型名称、配置参数和环境变量之间的对应关系
- 总控与子智能体之间如何完成编排与职责分工

本文覆盖的智能体包括：

- orchestrator
- director
- visual_director
- image_gen
- video_gen
- vision_analyzer

---

## 2. 总体架构结论

系统采用“**统一对话模型 + 按工具分流的媒体/视觉模型**”架构。

这意味着：

1. 所有 Agent 都通过 `veadk.Agent(...)` 直接实例化，并在构造时**显式传入模型配置参数**。
2. Agent 自身的自然语言理解、任务规划、工具决策与结果汇总，统一依赖 `.env` 驱动的主对话模型配置。
3. 当 Agent 调用图片生成、视频生成、视觉分析或部分结构化创作工具时，真正执行这些能力的底层模型由工具层进一步决定。
4. 因此，系统中存在两层“模型使用面”：
   - **Agent 层模型**：负责对话、调度、推理、转交、总结
   - **工具层模型**：负责生图、生视频、视觉理解，以及部分 JSON 化创作

从实现上看，这是一套“**显式 Agent 模型配置 + 按工具能力路由到底层媒体模型/服务提供方**”的架构。默认情况下所有智能体共享同一套全局 Agent 模型配置，但也支持通过 `.env` 对单个 Agent 做定向覆盖。

---

## 3. 初始化与运行链路

### 3.1 启动入口

系统从 CLI 入口启动，初始化顺序如下：

1. 先加载项目根目录下的 `.env`
2. 再导入 VeADK 运行时与总控智能体
3. 使用 `Runner` 挂载 `orchestrator_agent`
4. 后续所有用户输入都先交给总控智能体
5. 总控智能体根据阶段和任务内容，再决定是否转交给子智能体或调用工具

### 3.2 Runner 层配置

运行时主要配置包括：

- `agent=orchestrator_agent`
- `app_name="purevis_app"`
- `user_id="user_01"`
- 对话执行时额外传入 `RunConfig(max_llm_calls=30, streaming_mode=SSE)`

这里的 `RunConfig` 并不决定底层使用哪个模型，它主要控制：

- 单轮对话中最多允许多少次 LLM 调用
- 是否启用流式返回

### 3.3 Agent 实例化方式

所有智能体都在模块导入阶段直接完成实例化，典型结构一致：

- `name`
- `description`
- `instruction`
- `tools`
- 部分 Agent 额外包含 `sub_agents`

当前实现具备以下特征：

- 没有 Agent 工厂函数
- 没有延迟创建
- 所有 Agent 都通过统一辅助函数显式注入 `model_name`、`model_provider`、`model_api_base`
- 如果存在单 Agent 覆盖变量，则允许为某个 Agent 定向覆盖 `api_key` 与 `extra_config`

因此，**Agent 自身的模型来源已经从“隐式运行时默认值”升级为“显式 `.env` 配置 + 统一辅助函数注入”**。

## 4. 配置加载机制

### 4.1 主对话模型配置

主对话模型通过环境变量驱动，关键变量如下：

- `MODEL_AGENT_PROVIDER`
- `MODEL_AGENT_NAME`
- `MODEL_AGENT_API_BASE`
- `MODEL_AGENT_API_KEY`
- `MODEL_AGENT_EXTRA_CONFIG`

其中：

- `MODEL_AGENT_PROVIDER` 默认推荐使用 `openai`
- `MODEL_AGENT_NAME` 用于系统主对话、调度、写剧本、总结等文本任务
- `MODEL_AGENT_API_BASE` 推荐使用火山方舟 `api/v3` 端点
- `.example_env` 给出的默认示例是 `doubao-seed-2-0-pro-260215`

这套配置主要服务于 **VeADK Agent 对话层**。在当前项目中，各 Agent 会通过统一辅助函数显式读取这些值并注入构造参数。

### 4.1.1 单 Agent 覆盖机制

为兼顾一致性与灵活性，当前项目还支持对单个 Agent 做定向覆盖。命名规则如下：

- `MODEL_<AGENT_NAME>_PROVIDER`
- `MODEL_<AGENT_NAME>_NAME`
- `MODEL_<AGENT_NAME>_API_BASE`
- `MODEL_<AGENT_NAME>_API_KEY`
- `MODEL_<AGENT_NAME>_EXTRA_CONFIG`

例如：

- `MODEL_ORCHESTRATOR_NAME`
- `MODEL_VISUAL_DIRECTOR_API_BASE`
- `MODEL_IMAGE_GEN_EXTRA_CONFIG`

优先级顺序为：

1. 单 Agent 覆盖变量
2. 全局 `MODEL_AGENT_*`
3. 项目内置默认值

### 4.2 媒体提供方配置

图片、视频、视觉分析以及部分结构化设计工具，不直接绑定在 Agent 构造中，而是通过媒体提供方注册机制动态选择：

- `MEDIA_PROVIDER=auto` 时：
  - 若存在 `PUREVIS_API_KEY`，优先走 `purevis`
  - 否则若存在 `VOLCENGINE_ARK_API_KEY`，回退到 `volcengine_ark`
  - 兼容场景下，也支持回退到 `MODEL_TOOL_VISION_ANALYZER_API_KEY` 或 `MODEL_AGENT_API_KEY`
- 若显式设置了 `MEDIA_PROVIDER`，则按配置强制选择

这意味着同一个 `image_gen` 或 `video_gen` 智能体，在不同环境下可能落到不同底层服务。

### 4.3 Volcengine ARK 侧的工具层模型配置

当前项目已经把工具层能力拆成三路配置：

- 普通文本工具模型：
  - 使用 `MODEL_TOOL_TEXT_*`
  - 如果未配置，则自动继承 `MODEL_AGENT_*`
  - 适用于角色/场景/道具设计、分镜拆解、关键帧提示词、视频提示词等结构化文本工具
- 图片分析模型：
  - 使用 `MODEL_TOOL_VISION_ANALYZER_*`
  - 默认固定为 `doubao-seed-2-0-pro-260215`
  - 不继承 `MODEL_AGENT_*`
  - 适用于 `analyze_image`
- 生图模型：`IMAGE_MODEL_NAME`
  - 固定值：`doubao-seedream-4-5-251128`
- 生视频模型：`VIDEO_MODEL_NAME`
  - 固定值：`doubao-seedance-1-5-pro-251215`
- Volcengine ARK 媒体生成凭证：
  - 优先使用 `VOLCENGINE_ARK_API_BASE` / `VOLCENGINE_ARK_API_KEY`
  - 兼容场景下可回退到 `MODEL_TOOL_VISION_ANALYZER_API_BASE` / `MODEL_TOOL_VISION_ANALYZER_API_KEY`
  - 如果仍未配置，再回退到 `ARK_API_BASE` / `MODEL_AGENT_API_KEY`

因此，系统在 Volcengine 路径下实际上同时维护了多种模型能力：

- 主对话文本模型
- 工具层文本模型
- 工具层视觉模型
- 工具层图片生成模型
- 工具层视频生成模型

## 5. 依赖注入机制

系统的依赖注入主要有四类。

### 5.1 子智能体注入

总控智能体在初始化时直接持有五个子智能体实例：

- director_agent
- visual_director_agent
- image_gen_agent
- video_gen_agent
- vision_analyzer_agent

这是一种显式静态装配，不依赖运行时发现或反射注册。

### 5.2 工具注入

每个智能体通过 `tools=[...]` 声明其能力边界。  
这种方式决定了“谁能做什么”，也形成了清晰的职责隔离。

例如：

- 总控负责状态管理、风格治理、项目调度
- image_gen 负责提交生成任务、等待完成、下载、预览、状态登记
- video_gen 负责视频提交与落盘
- vision_analyzer 负责异步审图与分析

### 5.3 配置注入

项目状态与风格并不是在 Agent 构造时写死，而是在执行期通过工具加载：

- `get_project_state` 负责注入项目级配置，如画幅比例
- `get_prompt_style_context_state` 负责注入分媒介风格块，如 image / keyframe / video
- 风格治理类工具负责风格家族、子类、版本和回滚管理

### 5.4 全局提示注入

所有 Agent 的系统指令末尾都拼接了统一的资产保存规范。  
这相当于一层共享系统约束，用于确保所有生成物都落在统一的项目目录结构中。

## 6. 各智能体详细说明

## 6.1 orchestrator

### 职责定位

总控智能体，负责整个短剧工作流的阶段推进、状态维护、风格治理、任务分发与转交。

### Agent 层模型类型

- 类型：统一主对话文本模型
- 具体模型名：默认来自 `MODEL_AGENT_NAME`
- 实际来源：由统一辅助函数显式注入；如存在 `MODEL_ORCHESTRATOR_*` 则优先使用该组配置

### 初始化参数

该智能体实例化时包含以下核心参数：

- `name="orchestrator"`
- `description`
- 超长 `instruction`
- `sub_agents`
- `tools`

### 模型加载方式

不在构造参数中传入模型客户端对象，但会显式传入模型配置字段。  
默认情况下使用全局 Agent 模型配置；如存在单 Agent 覆盖变量，则以覆盖值为准。

### 依赖关系

#### 子智能体依赖

- director
- visual_director
- image_gen
- video_gen
- vision_analyzer

#### 工具依赖

- 项目状态创建、查询、剧集注册、主体发现
- 风格家族查询、风格配置读取/写入/回滚
- 项目仪表盘打开
- 目录读取

### 技术特征

- orchestrator 自身不直接持有生图或生视频工具
- 它的核心价值是“编排”和“转交”
- 它通过系统提示强制约束阶段顺序、命名规范、风格治理流程和失败回退策略

### 判断结论

orchestrator 是纯编排型 Agent，不直接绑定媒体模型。

---

## 6.2 director

### 职责定位

负责角色、场景、道具设计，剧本拆解，以及关键帧/视频提示词生成。

### Agent 层模型类型

- 类型：统一主对话文本模型
- 作用：理解用户意图、决定调用哪个创作工具、整理输出

### 工具层模型类型

当底层提供方为 `volcengine_ark` 时，以下创作工具最终会调用工具层文本模型生成结构化 JSON：

- `design_character`
- `design_scene`
- `design_prop`
- `breakdown_storyboard`
- `generate_keyframe_prompts`
- `generate_video_prompts`

其底层调用特点：

- 通过 `local_chat_completions(...)` 发起文本生成
- temperature 按工具分层配置，不再统一写死
- 默认值分别为：角色 `0.75`、场景 `0.65`、道具 `0.60`、分镜拆解 `0.40`、关键帧提示词 `0.45`、视频提示词 `0.45`
- 默认模型来自 `MODEL_TOOL_TEXT_*`
- 如果 `MODEL_TOOL_TEXT_*` 未配置，则自动继承 `MODEL_AGENT_*`

### 初始化参数

- `name="director"`
- `description`
- `instruction`
- `tools`

### 模型加载方式

director 本身不显式声明模型；  
真正执行结构化创作时，由工具层发起二次文本模型调用。

### 依赖关系

- 依赖设计类工具完成设定生成
- 依赖分镜拆解工具输出结构化片段
- 依赖文件工具保存策划、剧本和分镜文本

### 判断结论

director 是“文本 Agent + 文本模型工具”的双层架构，不是单一模型直出。

---

## 6.3 visual_director

### 职责定位

负责将参考图或剧情梗概扩展为电影级关键帧序列，并输出 Exhausted JSON。

### Agent 层模型类型

- 类型：统一主对话文本模型
- 特征：偏复杂长提示词推理

### 工具层模型类型

无独立图片生成或视频生成模型参与。  
它调用的工具主要用于：

- 读取项目状态
- 注入关键帧风格上下文
- 查询目录
- 保存 markdown 结果

### 初始化参数

- `name="visual_director"`
- `description`
- `instruction=VISUAL_DIRECTOR_PROMPT + 全局资产规范`
- `tools=[save_text_file, read_text_file, list_directory, get_project_state, get_prompt_style_context_state]`

### 模型加载方式

完全依赖主对话模型进行推理和结构化输出。  
工具只负责补充上下文和保存结果，不承担媒体生成任务。

### 依赖关系

- 依赖项目状态工具读取 `aspect_ratio` 等设定
- 依赖风格上下文工具注入 `keyframe` 风格要求
- 依赖文件工具保存分镜文档

### 判断结论

visual_director 是当前系统中最纯的“文本推理型智能体”，没有单独的媒体模型实例化逻辑。

---

## 6.4 image_gen

### 职责定位

负责文生图、图生图、多视图、表情表、姿势表，以及生成完成后的下载、预览、状态登记。

### Agent 层模型类型

- 类型：统一主对话文本模型
- 作用：判断使用哪种生成工具、何时采用图生图、如何组织闭环执行

### 工具层模型类型

#### 在 Volcengine 路径下

- 图片生成模型：`doubao-seedream-4-5-251128`

该模型在提交图片生成请求时被显式写入请求载荷：

- `model=IMAGE_MODEL_NAME`

#### 在 PureVis 路径下

- 由 PureVis 提供图片生成能力
- Agent 无需感知具体模型名称，只通过 provider 接口统一调用

### 关键执行参数

image_gen 的执行依赖以下关键输入：

- `prompt`
- `aspect_ratio`
- `input_images` 或 `ref_image`

其中：

- `aspect_ratio` 来自项目状态
- 风格约束来自 `get_prompt_style_context_state(project_name, "image")`
- 若是图生图，则本地图片路径会被工具层自动编码并转给底层 API

### 初始化参数

- `name="image_gen"`
- `description`
- 闭环执行导向的 `instruction`
- 16 个工具

### 模型加载方式

image_gen 本身不直接创建图像模型实例。  
实际流程为：

1. Agent 决定调用哪种图像工具
2. 工具通过 provider 选择具体服务
3. 若走 Volcengine，则调用固定的 Seedream 生图模型
4. 结果返回后，Agent 继续完成等待、下载、预览和登记

### 依赖关系

- 项目状态读取
- 图片风格上下文注入
- 媒体生成工具
- 任务等待工具
- 下载与本地预览工具
- 主体状态登记工具

### 判断结论

image_gen 属于典型的“文本编排 + 图片模型执行”型智能体。

---

## 6.5 video_gen

### 职责定位

负责将关键帧图片与视频提示词转换为视频片段，并保存到项目输出目录。

### Agent 层模型类型

- 类型：统一主对话文本模型
- 作用：组织视频生成任务、整合风格约束、控制工具执行顺序

### 工具层模型类型

#### 在 Volcengine 路径下

- 视频生成模型：`doubao-seedance-1-5-pro-251215`

提交请求时写入的关键参数包括：

- `model=VIDEO_MODEL_NAME`
- `ratio=aspect_ratio`
- `resolution="720p"`
- `duration`
- `generate_audio`

#### 在 PureVis 路径下

- 由 PureVis 提供视频生成服务

### 初始化参数

- `name="video_gen"`
- `description`
- `instruction`
- `tools`

### 模型加载方式

video_gen 自身不实例化视频模型；  
它通过统一工具接口把任务下发给底层 provider，再由 provider 决定调用哪个服务。

### 依赖关系

- `get_project_state` 注入画幅配置
- `get_prompt_style_context_state(project_name, "video")` 注入视频风格块
- `generate_video` 提交生成
- `wait_for_task` 等待完成
- `download_and_save_media` 保存输出

### 判断结论

video_gen 是文本编排层与视频模型执行层分离最清晰的智能体之一。

---

## 6.6 vision_analyzer

### 职责定位

负责对图片进行审美评估、角色/场景一致性分析、修改建议输出与是否通过的判定。

### Agent 层模型类型

- 类型：统一主对话文本模型
- 作用：组织分析流程、解释返回结果、形成可读结论

### 工具层模型类型

#### 在 Volcengine 路径下

- 视觉理解模型：默认 `doubao-seed-2-0-pro-260215`
- 实际取值来源：`MODEL_TOOL_VISION_ANALYZER_*`
- 默认不继承 `MODEL_AGENT_*`

工具层调用特点：

- 通过 `local_chat_completions(..., model=get_tool_vision_model_name(), is_vision=True)` 发起视觉请求
- 默认 `temperature=0.15`
- 可由 `MODEL_TOOL_VISION_ANALYZER_TEMPERATURE` 覆盖
- 请求会将本地图片路径转换为 Data URI 后提交

#### 在 PureVis 路径下

- 使用 PureVis 的视觉分析能力

### 初始化参数

- `name="vision_analyzer"`
- `description`
- `instruction`
- `tools`

### 模型加载方式

该智能体本身不持有视觉模型对象。  
其工作流是：

1. Agent 决定分析类型和分析目标
2. 调用 `analyze_image`
3. 底层 provider 选择视觉模型并执行
4. 返回结构化结果后，Agent 再产出审美和一致性结论

### 依赖关系

- 图片分析工具
- 异步等待工具
- 状态查询工具
- 文本读取工具
- 本地图片预览工具

### 判断结论

vision_analyzer 本质上是“视觉模型分析 + 文本模型解释”的两段式结构。

## 7. 模型映射总表

| 智能体 | Agent 层模型 | 工具层模型/服务 | 是否显式在 Agent 构造中配置模型 |
| --- | --- | --- | --- |
| orchestrator | 显式注入的 Agent 文本模型 | 无直接媒体模型 | 是 |

| video_gen | 显式注入的 Agent 文本模型 | 生视频模型 / PureVis 视频服务 | 是 |
| vision_analyzer | 显式注入的 Agent 文本模型 | 视觉模型 / PureVis 视觉分析服务 | 是 |

- `MODEL_AGENT_EXTRA_CONFIG`

## 9. 设计上的重要特征

### 9.1 模型配置是“分层的”

系统不是把所有能力都压在一个模型上，而是按层次拆分：

- Agent 层负责语言规划与调度
- 工具层负责垂直能力执行

这使得系统更适合多模态工作流，也更方便后续替换某一类底层服务。

### 9.2 同名“文本模型”可能出现双配置

当前实现里存在两套文本模型入口：

- Agent 运行时使用的主对话模型入口
- 工具层 `MODEL_TOOL_TEXT_*` 使用的结构化创作模型入口

默认情况下两者都指向同一个推荐模型，但从设计上说，它们并不是强绑定的。

### 9.3 Agent 配置显式化，但默认仍保持统一

当前项目已经做到：

- 每个 Agent 在构造时显式传入模型配置
- 支持按 Agent 单独覆盖模型名称、Provider、API Base、API Key、额外参数
- 默认仍通过全局 `MODEL_AGENT_*` 保持统一配置

这说明当前架构在保留“统一调度层 + 差异化工具层”的同时，也增强了模型配置的可审计性与可维护性。

## 10. 维护建议

1. 如果未来需要提升某个 Agent 的专业性，优先考虑在工具层增加更强的专用模型，而不是先拆分 Agent 层对话模型。
2. 如果希望保持行为一致，建议让主对话模型与工具层文本模型保持同代系。
3. 当前已具备每 Agent 独立模型覆盖能力；如需启用，请优先通过 `.env` 的单 Agent 覆盖变量管理，而不是在各文件中散落硬编码。
4. 若要排查输出风格不一致问题，应优先检查：
   - `.env` 中的主文本模型配置
   - `MEDIA_PROVIDER` 是否切换了底层服务
   - 工具层文本模型、视觉模型是否被单独覆盖
   - 项目状态中的画幅与结构化风格配置是否注入成功

---

## 11. 最终结论

本系统中的六个智能体现在都采用显式 Agent 模型配置：每个 Agent 在构造时都会注入 `model_name`、`model_provider`、`model_api_base` 等字段，并默认共享一套全局 `.env` 配置；如有需要，也可以对单个 Agent 单独覆盖。同时，在执行图片、视频、视觉分析等任务时，仍然通过工具层路由到不同的专用模型或服务：

- 总控负责调度，不直接使用媒体模型
- 导演负责创作，可通过工具层二次调用文本模型
- 视觉导演是纯文本推理型智能体
- 生图智能体依赖图片生成模型
- 生视频智能体依赖视频生成模型
- 视觉分析智能体依赖视觉理解模型

从维护角度看，这是一套“**统一智能体外壳 + 专用能力工具化**”的多智能体系统。
